Checkpoints

Checkpoints give you a safety net for every agent task. Before the first write operation in a task, Obsilo automatically takes a snapshot of all tracked files. If something goes wrong, you can restore to that snapshot with one click.

How It Works

Obsilo maintains a shadow git repository at:

.obsidian/plugins/obsilo-agent/checkpoints/

This is separate from your vault's own git history (if you use git for your vault). It uses isomorphic-git — a pure JavaScript git implementation that requires no native git binary.

Checkpoint Lifecycle

  1. Task starts — you send a message and the agent begins working
  2. First write detected — before executing the first write_file, edit_file, delete_file, or similar operation, Obsilo takes a snapshot of all currently-tracked files
  3. Task completes — the checkpoint is retained and listed in Settings → Backup
  4. Auto-cleanup (if enabled) — old checkpoints beyond the retention limit are removed

What Gets Tracked

The shadow repo tracks all files that the agent has read or written in previous tasks. On the first snapshot, it also includes all files in the vault root (excluding ignored paths). This means:

  • Any file the agent has touched can be restored
  • Files in .obsidian-agentignore are not tracked
  • The checkpoint doesn't restore files to a vault-wide snapshot — it restores only the files changed in that task

Restoring from a Checkpoint

  1. Go to Settings → Advanced → Backup
  2. You'll see a list of checkpoints, ordered by date, with the task that created them
  3. Click Preview on a checkpoint to see a diff of what changed
  4. Click Restore to revert the files changed in that task back to their pre-task state
Restore only affects changed files Restoring a checkpoint reverts only the files that were modified in that specific task — not the entire vault. Files created by the task are moved to trash; files edited are restored to their pre-edit content.

Configuration

In Settings → Vault → Checkpoints:

SettingDefaultDescription
Enable CheckpointsOnMaster toggle. When off, no snapshots are taken and restore is unavailable.
Checkpoint Timeout30 secondsMaximum time to wait for a snapshot to complete. If exceeded, the task continues without a checkpoint.
Auto-cleanupOnAutomatically remove old checkpoints to save disk space.

Checkpoints vs. Your Vault's Git

If you already use git for your vault, you might wonder why Obsilo has its own checkpoint system. The reasons:

  • Non-invasive: Checkpoints don't pollute your vault's git history with agent commits
  • No git required: Works even if you don't use git for your vault at all
  • Automatic: Happens without any user action — no need to remember to commit before each agent task
  • Scoped: Restores only the files a specific task touched, not the whole vault
  • Task-linked: Each checkpoint is linked to the conversation that created it, making it easy to understand what changed and why

Disk Usage

The shadow repo uses standard git compression. For most vaults, each checkpoint adds a few kilobytes. With auto-cleanup enabled, old checkpoints are pruned automatically. You can also manually clear checkpoints from Settings → Backup.
