MCP Client

The McpClient manages connections to external MCP (Model Context Protocol) servers, providing the agent with access to tools beyond the built-in set. It supports three transport mechanisms and includes security validation for command execution.

Architecture

The MCP client follows a lean design: no OAuth, no file-watching, no auto-reconnect. Each configured server gets exactly one SDK Client instance from @modelcontextprotocol/sdk. Connections are managed explicitly through connect/disconnect calls rather than reactive file watchers.

McpClient | +-- Map<serverId, McpConnection> | | | +-- status: 'connected' | 'connecting' | 'disconnected' | 'error' | +-- client: MCP SDK Client | +-- tools: McpToolInfo[] | +-- connect(serverConfig) --------> Transport Layer +-- connectAll(configs[]) --------> Parallel connections +-- callTool(server, tool, args) -> Tool execution +-- disconnect(serverId) ---------> Cleanup

Transport Configuration

Each server entry in settings specifies a transport type and its connection parameters. Below are concrete configuration examples for each supported transport.

stdio — spawns a child process and communicates via stdin/stdout:

{
  transport: 'stdio',
  command: 'npx',
  args: ['-y', '@modelcontextprotocol/server-filesystem', '/path/to/dir']
}

sse — connects to a remote server via HTTP Server-Sent Events:

{
  transport: 'sse',
  url: 'http://localhost:3000/sse',
  headers: { 'Authorization': 'Bearer ...' }
}

streamable-http — uses the newer MCP streaming HTTP protocol:

{
  transport: 'streamable-http',
  url: 'http://localhost:3000/mcp',
  headers: {}
}

Transport Types

TransportProtocolUse Case
stdioStdin/stdout of a child processLocal CLI tools (e.g., filesystem, database)

Command Validation

For stdio transport, the command string is validated against shell metacharacters to prevent injection:

// Blocked characters:
const DANGEROUS = /[;&|`$(){}[\]<>\\]/;
// If command matches, connection is rejected

This prevents commands like npx some-tool; rm -rf / from executing.

Connection Flow

  1. Validate config — Check transport type, validate command (stdio) or URL (sse/http).
  2. Create transport — Instantiate the appropriate MCP SDK transport.
  3. Connect with timeout — Race connection against a configurable timeout (default: 60 seconds).
  4. Discover tools — Call listTools() on the connected server to get available tools.
  5. Store connection — Save the connection with status, client, and discovered tools.

connectAll() runs all server connections in parallel via Promise.all(), so slow or failing servers do not block others. The default timeout is 60 seconds per connection. disconnect(serverId) calls client.close() on the SDK instance and clears all internal references (client, tools, status).

UseMcpToolTool (LLM Bridge)

The use_mcp_tool tool bridges LLM tool calls to MCP server invocations:

// Tool parameters:
{
  server_name: string,  // MCP server identifier
  tool_name: string,    // Tool on that server
  arguments?: object   // Tool-specific input
}

The tool sets isWriteOperation = true as a conservative default — all MCP tool calls are treated as write operations unless the name proves otherwise. Write detection uses keyword matching: if the tool name contains any of write, create, delete, update, edit, modify, remove, insert, or set, the call requires user approval before execution.

Per-Mode Server Whitelist

Each mode can define which MCP servers are accessible via settings.activeMcpServers. An empty array means all servers are accessible. Custom modes can restrict access to specific servers via mcpWhitelist.

Error Handling

ScenarioBehavior

Tool call errorReturns 'Error calling {tool} on {server}: {message}'

Introspection API

The McpClient exposes several methods for querying the current state of MCP connections:

  • getConnections() — returns all connection entries regardless of their status (connected, connecting, disconnected, error). Useful for displaying the full server list in the UI.
  • getConnection(name) — returns a single connection entry by server name, or undefined if not found.
  • getAllTools() — returns a flat list of all discovered tools from connected servers only. Disconnected or errored servers are excluded from the tool list.

Lifecycle

The MCP client follows the plugin lifecycle:

  1. Startup — mcpClient.connectAll(settings.mcpServers) is called during plugin initialization. All configured servers connect in parallel.
  2. Tool discovery — Each successfully connected server reports its tools via client.listTools(). These are cached in the connection entry.
  3. Runtime — Tools are available for the agent to invoke via use_mcp_tool. Connection status is checked before each call.
  4. Plugin unload — mcpClient.disconnectAll() is called, closing all SDK clients and clearing all connection state.
For how to configure MCP servers, see the MCP Servers user guide.
