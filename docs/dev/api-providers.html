API Providers

Obsilo uses an adapter pattern to support multiple LLM providers through a unified ApiHandler interface. Two concrete implementations handle all 7 provider types: AnthropicProvider for the Anthropic API, and OpenAiProvider for all OpenAI-compatible APIs.

ApiHandler Interface

interface ApiHandler {
  createMessage(
    systemPrompt: string,
    messages: MessageParam[],
    tools: ToolDefinition[]
  ): AsyncGenerator<StreamChunk>;

  getModel(): string;
}

Stream Chunk Types

Both providers normalize their streams into 4 chunk types:

TypeContentWhen
textIncremental text tokensLLM generates natural language
thinkingExtended thinking contentAnthropic models with thinking enabled

Provider Types

AnthropicProvider

Uses the official @anthropic-ai/sdk with dangerouslyAllowBrowser: true -- this flag is required because Obsilo runs inside the Electron renderer process, not a Node.js server environment.

  • Streaming: Native SDK streaming with content block events (content_block_start, content_block_delta, content_block_stop).
  • Tool accumulation: A Map<number, { id, name, inputJson }> keyed by content block index. Each content_block_delta of type input_json_delta appends to the inputJson string. The full JSON is parsed on the content_block_stop event, and the complete tool call is yielded as a single tool_use chunk.
  • Thinking blocks: Tracked in a separate Map<number, { text }>. Each thinking delta is yielded incrementally as a thinking chunk so the UI can display reasoning in real time.
  • Usage tracking: inputTokens are extracted from the message_start event, outputTokens from the message_delta event. Both are yielded as a final usage chunk.
  • Temperature: Default 0.2, clamped to max 1.0.
  • Token limit: max_tokens parameter.

OpenAiProvider

Covers 6 OpenAI-compatible provider types with different base URLs and authentication:

ProviderBase URLAuth
OpenAIhttps://api.openai.com/v1Bearer token
Ollamahttp://localhost:11434/v1None
LM Studiohttp://localhost:1234/v1None
OpenRouterhttps://openrouter.ai/api/v1Bearer + HTTP-Referer + X-Title
AzureCustom deployment URLapi-key header + api-version query
CustomUser-configured URLBearer token
  • SSE stream parsing: Manual parsing of data: lines from the raw response stream. Each line is trimmed and checked for the [DONE] sentinel. Parsed JSON chunks are processed to extract delta content.
  • Tool accumulation: A Map<number, ToolCallAccumulator> collects streaming function call fragments. Each SSE delta may contain a partial function name or argument JSON. The accumulated tool call is finalized and yielded as a complete tool_use chunk when the stream moves to the next tool or ends.
  • Temperature: Omitted entirely for O-series models (they do not support the parameter). Azure default temperature is also omitted to use the deployment's configured default.
  • Token limit: Azure uses max_completion_tokens instead of max_tokens. All other providers use max_tokens.
  • Message conversion: Internally all messages use Anthropic's content block format. The OpenAiProvider converts these to OpenAI's expected format: Anthropic tool_use content blocks become OpenAI tool_calls on the assistant message, and Anthropic tool_result blocks become separate messages with role: 'tool'.
  • Auth specifics: Azure uses the api-key HTTP header (not Bearer). OpenRouter adds HTTP-Referer and X-Title headers for attribution tracking. Local providers (Ollama, LM Studio) send no authentication.

Internal Message Format

All providers normalize to Anthropic's content block format internally:

interface MessageParam {
  role: 'user' | 'assistant';
  content: string | ContentBlock[];
}

// ContentBlock types: text, image, tool_use, tool_result

Provider Factory

buildApiHandlerForModel() reads the model configuration to determine the provider type and instantiates the appropriate handler. Each model can have custom temperature, API version, and base URL overrides.

Custom Model Configuration

Every model -- both built-in and user-defined -- is represented by the CustomModel interface:

interface CustomModel {
  name: string;           // Model ID for API calls (e.g. "claude-sonnet-4-20250514")
  provider: ProviderType; // 'anthropic' | 'openai' | 'ollama' | 'lmstudio' | 'openrouter' | 'azure' | 'custom'
  displayName?: string;
  apiKey?: string;
  baseUrl?: string;
  enabled: boolean;
  isBuiltIn?: boolean;
  maxTokens?: number;
  temperature?: number;
  apiVersion?: string;    // Azure only
}

Built-in models come pre-configured with sensible defaults (provider endpoint, max tokens, temperature) and have isBuiltIn: true. Users can add custom models through the settings UI to connect to any OpenAI-compatible endpoint or to override defaults on built-in models. The provider field determines which handler class is instantiated by the factory.

Error Handling

Both providers implement defensive error handling for the inherently unreliable nature of streaming LLM APIs:

  • Stream interrupts: SSE connection drops (network timeouts, server errors) are caught at the stream reader level. Any partial content accumulated before the interruption is preserved and yielded so the user sees what was generated before the failure.
  • Rate limiting: Provider-specific Retry-After headers are respected. When a 429 response is received, the provider waits the indicated duration before retrying.
  • Tool JSON parse errors: Because tool call arguments arrive as incremental JSON fragments, the accumulated string may be malformed if the stream is interrupted. Parsing is performed inside a try-catch on the content_block_stop event (Anthropic) or stream completion (OpenAI). Malformed tool calls are dropped with a warning rather than crashing the agent loop.
  • Abort signal: An AbortController is passed to all API calls. When the user clicks "Stop" in the UI, the controller's abort() is called, which cancels the in-flight HTTP request and terminates the async generator mid-stream.
For how to configure providers and models, see Providers & Models.
