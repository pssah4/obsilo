Governance & Safety

Obsilo implements defense in depth with multiple security layers: path-based access control, user approval, git checkpoints, audit logging, SafeStorage (OS keychain encryption for API keys), sub-agent depth guard, content truncation, and tool repetition detection.

IgnoreService

Path-based access control using .obsidian-agentignore and .obsidian-agentprotected files.

Hard-Coded Blocks

PathReason
.git/Git internals
.obsidian/workspaceObsidian workspace state
.obsidian/workspace.jsonObsidian workspace state
.obsidian/cacheObsidian cache

Pattern Syntax

Gitignore-style. * matches non-/, ** matches anything, trailing / = directory.

ReDoS protection: reject patterns > 200 chars or with 3+ consecutive **.

Fail-closed: all paths inaccessible until governance rules load.

Example Configuration

# .obsidian-agentignore
# Ignore private journal
journal/private/
*.secret.md

# .obsidian-agentprotected (read-only access)
templates/**

Files matching .obsidian-agentignore patterns are completely invisible to the agent — no read, no write, no listing. Files matching .obsidian-agentprotected patterns can be read but not modified or deleted. Both files are placed at the vault root and follow the same glob syntax.

Auto-Approval System

Per-group approval toggles. Default: master toggle disabled.

Agent tools always auto-approved: ask_followup_question, attempt_completion, update_todo_list, open_note, update_settings, configure_model.

AutoApprovalConfig Interface

interface AutoApprovalConfig {
  enabled: boolean;        // Master toggle
  read: boolean;           // Read operations
  noteEdits: boolean;      // apply_diff, insert_content
  vaultChanges: boolean;   // write_to_file, move_file, delete_file, create_directory
  web: boolean;            // web_search, web_fetch
  mcp: boolean;            // use_mcp_tool
  mode: boolean;           // switch_mode
  subtasks: boolean;       // new_task
  question: boolean;       // ask_followup_question
  todo: boolean;           // update_todo_list
  skills: boolean;         // execute_command, enable_plugin
  pluginApiRead: boolean;  // call_plugin_api (read)
  pluginApiWrite: boolean; // call_plugin_api (write)
  recipes: boolean;        // execute_recipe
}

Defaults: enabled=false, read=true, all write groups =false. The system is fail-closed: if no approval callback is registered on the execution context, write operations are denied outright rather than silently executed.

Approval Decision Flow

The following decision tree shows how the pipeline resolves whether a tool call requires user approval or can execute automatically.

Tool call arrives | Is tool in the always-auto-approved set? (agent group) | | YES --> Execute NO | Is auto-approval master toggle enabled? | | NO --> Show approval UI YES | Is group toggle enabled for this tool? | | NO --> Show approval UI YES --> Execute

When the approval UI is shown, the user can approve or deny the operation. Denied operations return a rejection message to the agent, allowing it to adjust its approach. Approved operations continue to the next pipeline stage (checkpoint creation, then execution).

Git Checkpoints

GitCheckpointService uses isomorphic-git to maintain a shadow git repository. Before every write operation, a snapshot is created. Users can restore to any checkpoint from the UI.

  • Pure-JS — no system git required
  • Shadow repo — .obsidian/plugins/obsilo-agent/checkpoints/
  • Auto-snapshot — before every write operation

CheckpointInfo Interface

interface CheckpointInfo {
  taskId: string;
  commitOid: string;
  timestamp: number;
  filesChanged: string[];
  toolName: string;
  newFiles: string[];
}

Storage & Restore

Checkpoints are held in an in-memory Map<taskId, CheckpointInfo[]> for fast lookup during an active session. Each task maintains its own ordered list of checkpoints.

Restore process: For existing files, git.readBlob() reads the file content from the checkpoint commit. For files that were created after the checkpoint (tracked in newFiles), vault.delete() removes them from the vault entirely.

Post-restart recovery: Since the in-memory map is lost on plugin reload, the service falls back to a git.log() scan of the shadow repo to reconstruct checkpoint history from commit metadata.

Operation Logger

Every tool execution is logged to a JSONL file for full audit traceability.

LogEntry Structure

{
  timestamp: string;    // ISO 8601
  taskId: string;
  mode: string;
  tool: string;
  params: object;       // Sanitized input parameters
  result: string;       // Sanitized output summary
  success: boolean;
  durationMs: number;
  error?: string;       // Present only on failure
}

Sanitization Rules

Before writing to disk, all log entries are sanitized to prevent accidental secret exposure:

  • Fields named password, token, api_key, secret, key, or auth are replaced with [REDACTED].
  • Content fields (note bodies, file contents) are summarized as "[N chars]" to reduce log volume.
  • URLs are stripped of query parameters to avoid leaking tokens embedded in URLs.
  • Any string value exceeding 500 characters is truncated with an ellipsis.

File Rotation

Logs are stored as daily files at .obsidian/plugins/obsilo-agent/logs/YYYY-MM-DD.jsonl. Files older than 30 days are automatically deleted on plugin startup. Each line in the JSONL file is a self-contained JSON object, making the logs easy to parse and filter with standard tooling.

Consecutive Mistake Limit

A safety valve that halts runaway agent loops. An internal counter tracks consecutive tool errors (any tool execution that returns success: false). The counter resets to zero on every successful tool call.

When the counter reaches the configured limit, the agent task is stopped immediately and the user is notified. This prevents scenarios where the agent repeatedly attempts a failing operation, burning tokens and potentially causing unintended side effects.

SettingDefaultDescription
advancedApi.consecutiveMistakeLimit3Number of consecutive tool errors before the task is halted

The limit is configurable per-session. Setting it to 0 disables the safety valve entirely (not recommended). The counter is scoped to the current task and is not shared across subtasks.

Defense in Depth

Layer 1: IgnoreService -- Path-based blocks (hard + configurable) Layer 2: Approval System -- User consent for write operations Layer 3: Git Checkpoints -- Undo capability for every write Layer 4: Operation Logger -- Full audit trail (JSONL)
For user-facing permission controls, see Permissions & Safety.
