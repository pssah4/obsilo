Semantic Search Pipeline

Obsilo implements hybrid search combining vector similarity (Vectra HNSW) with keyword matching (BM25/TF-IDF) via Reciprocal Rank Fusion, enhanced by HyDE expansion and wikilink graph augmentation.

Vectra LocalIndex

Pure-TypeScript HNSW vector index. Storage: .obsidian/plugins/obsilo-agent/semantic-index/. Requires a configured embedding model (e.g. OpenAI text-embedding-3-small or Ollama nomic-embed-text).

Embedding via API

Any OpenAI-compatible embedding endpoint. Requests are batched with a batch size of 16 embeddings per API call. Failures are handled with exponential backoff: 1s, 2s, 4s, 8s delays with a maximum of 4 retries. Each request has a 30-second timeout.

Model key format: {provider}:{modelName} (e.g. openai:text-embedding-3-small). Changing the model key triggers a full index rebuild since embedding dimensions and vector spaces differ between models.

Heading-Aware Chunking

5-stage chunking pipeline:

  1. Split by headings — break document at Markdown heading boundaries.
  2. Respect chunk size — default 2000 characters per chunk.
  3. 10% overlap — between consecutive chunks for context continuity.
  4. Hard split at 70% — to avoid mid-word breaks.
  5. Ellipsis prefix — for continued chunks.

YAML Frontmatter Handling

YAML frontmatter delimiters (---) are stripped, but the frontmatter content itself is preserved and prepended to the document body before chunking. This ensures metadata like tags, aliases, and descriptions contribute to the semantic representation of the note.

Single-chunk optimization: If the entire file content (after frontmatter processing) fits within the configured chunk size, the chunking pipeline is bypassed and the full text is used as a single chunk. This avoids unnecessary overlap and ellipsis artifacts on small files.

BM25/TF-IDF with Stemming

Tokenization: Text is lowercased and split on the regex pattern [\s\-_/,.;:!?()\[\]{}"'\`|@#=+*<>~^]+. Tokens shorter than 3 characters are filtered out to remove noise from short fragments and common articles.

Suffix stemmer with rules for:

  • English: -ing, -tion, -ness, -ment, -able, -ful, -less, -ous, -ive, -ly, -ed, -er, -est, -es, -s
  • German: -ung, -heit, -keit, -isch, -lich, -bar, -sam, -en, -er, -es, -te, -ten

Scoring: TF = matches/total tokens, IDF = log((N+1)/(df+1)) where N is total document count and df is documents containing the term. The +1 smoothing prevents division by zero and dampens extreme weights for rare terms. Final score: SUM(TF * IDF) across all query terms.

Per-file deduplication: When multiple chunks from the same file appear in results, only the highest-scoring chunk per file is retained. This prevents a single long document from dominating the result set.

Hybrid Search & RRF Fusion

User Query | +---> Semantic Search (Vectra HNSW, top_k * 3) | | | v | Rank List A (by cosine similarity) | +---> BM25 Keyword Search (TF-IDF, stemming) | | | v | Rank List B (by BM25 score) | +---> RRF Fusion: score(doc) = SUM(1 / (k + rank_i)), k=60 | | | v | Merged & deduplicated results | +---> Graph Augmentation: 1-hop wikilink expansion (max 5 per hit) | v Final results (top_k)

searchK Inflation

To ensure enough candidates survive deduplication and filtering, the internal search retrieves more results than the requested topK:

  • With metadata filters: min(topK * 4, 80) — extra inflation compensates for post-filter attrition.
  • Without filters: min(topK * 3, 40) — less inflation needed when no candidates are dropped.

Post-Search Metadata Filters

After RRF fusion, results can be further narrowed by metadata filters provided by the caller:

FilterMatch Logic
folderPrefix match on file path (e.g. projects/ matches all files under that folder)
tagsAny-match — result is included if it contains at least one of the specified tags
sinceFile modification time must be after the given timestamp

Source Labeling

Each result in the final set is labeled with its origin for transparency: 'semantic' if found only by vector search, 'keyword' if found only by BM25, or 'hybrid' if both retrieval paths returned the same document. This labeling helps consumers understand why a result was returned and diagnose search quality.

HyDE (Hypothetical Document Expansion)

When enabled, the LLM generates a hypothetical note excerpt — a 2-3 sentence passage that would plausibly answer the query if it existed in the vault. This synthetic text is then embedded alongside the original query vector. Both embeddings are used for the semantic search pass, effectively doubling the chance of matching relevant documents that use different vocabulary than the query.

HyDE is particularly effective for abstract or conceptual queries (e.g. "how do I organize my project notes?") where the user's phrasing may not overlap with the actual note content. For exact keyword lookups, the BM25 branch already covers the match.

Silent fallback: If the LLM call fails (timeout, rate limit, API error), HyDE is silently skipped and the search proceeds with only the original query embedding. No error is surfaced to the user.

Incremental Build

Index built incrementally: only new/modified files re-indexed. File modification tracked via a checkpoint file at {indexDir}/index-meta.json which records the last-indexed mtime for every file.

Rebuild Phases

  1. Phase A — Cleanup: For each modified or deleted file, delete all existing chunks belonging to that file from the Vectra index.
  2. Phase B — Index: Read the file, run the heading-aware chunker, batch-embed the chunks (up to embeddingBatchSize per API call), insert into the index, and commit progress every 20 files.

Cancellation: The build can be cancelled at any time (e.g. plugin unload or user action). Partial progress is saved — files that were already committed remain indexed. The next build resumes from where it left off.

Auto-Update Queue

File modifications detected by Obsidian's vault events are pushed into an auto-update queue backed by a Set<string> for path deduplication. The queue processes files one at a time (concurrency of 1) with a 2-second pause between files to avoid starving the main thread. Rapid successive edits to the same file are naturally deduplicated by the Set.

Auto-Trigger Mechanism

When semanticAutoIndexOnChange is enabled in settings, the plugin registers four Obsidian vault event listeners that keep the index current in real time:

Vault EventAction
modifySchedule file for re-indexing (2s debounce)
createSchedule file for indexing (2s debounce)
deleteRemove file from index immediately
renameRemove old path, schedule new path for indexing

Debounce Strategy

Each file path gets its own debounce timer (2 seconds) stored in a Map<string, Timer>. This prevents thrashing while the user is actively typing. When the timer fires, the file is pushed into the auto-update queue (queueAutoUpdate) rather than indexed directly, ensuring concurrency of 1 and preventing concurrent embedding API calls from freezing the main thread.

Vault Event (modify/create/rename) | scheduleFileIndex() [src/main.ts] | per-file debounce timer (2s) | excluded folders filtered out v queueAutoUpdate() [SemanticIndexService] | Set<string> for path dedup | concurrency: 1 | 2s pause between files v updateFile() | delete old chunks -> re-chunk -> embed -> insert v Index updated

Indexable File Types

Markdown files (.md) are always indexed. When indexPdfs is enabled, document files (.pdf, .pptx, .xlsx, .docx) are also indexed using the Document Parser Pipeline.

Configuration

The following parameters control index behavior. All are set in the plugin settings under the Semantic Search section.

ParameterDefaultDescription
batchSize20Number of files processed per index commit. Higher values reduce commit overhead but increase memory usage.
embeddingBatchSize16Number of embeddings per API call (API mode only). Balances throughput against rate limits.
chunkSize2000Maximum characters per chunk. Changing this value triggers a full index rebuild since chunk boundaries shift.
storageLocation'obsidian-sync'Index storage path strategy. 'obsidian-sync' stores inside the plugin data folder (syncs with Obsidian Sync).
indexPdfsfalseWhen enabled, PDF files in the vault are extracted and indexed alongside Markdown notes.
