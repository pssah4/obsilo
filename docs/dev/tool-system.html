Tool Execution Pipeline

Every tool call from the LLM passes through a 6-step governance pipeline before reaching the actual tool implementation. This ensures safety, auditability, and user control.

The 6-Step Pipeline

LLM returns tool_use block | [1] VALIDATE -----> Tool exists? Input valid? Mode allows it? | | | (reject if invalid) | [2] IGNORE CHECK --> IgnoreService: path blocked? | | | (deny if ignored/protected) | [3] APPROVAL ------> Write operation? Auto-approve config? | | | (show diff, wait for user) | [4] CHECKPOINT ----> GitCheckpointService: snapshot before write | [5] EXECUTE -------> BaseTool.execute(input, context) | [6] LOG ------------> OperationLogger: append to JSONL | v Return tool_result to LLM

BaseTool Interface

Every tool extends BaseTool:

abstract class BaseTool {
  abstract name: string;
  abstract isWriteOperation: boolean;
  abstract execute(
    input: Record<string, any>,
    context: ToolExecutionContext
  ): Promise<string>;

  formatSuccess(msg): string;  // wraps in <success> tags
  formatError(msg): string;    // wraps in <error> tags
  formatContent(msg): string;  // wraps in <content> tags
}

The isWriteOperation flag determines whether the tool needs approval and checkpoint creation.

Tool Metadata

The toolMetadata.ts file is the single source of truth for tool display information:

interface ToolMeta {
  group: 'read' | 'vault' | 'edit' | 'web' | 'agent' | 'mcp' | 'skill';
  label: string;
  description: string;
  icon: string;
  signature: string;
}

Groups map to UI badge colors: read vault edit web agent mcp

Tool Repetition Detection

A sliding window of the last 5 tool calls tracks tool name + arguments combinations. If the same combination appears 3 times, the loop is terminated to prevent infinite repetition.

For a user-friendly overview of all tools, see the Tools Reference.
