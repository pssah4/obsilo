Tool Execution Pipeline

Every tool call from the LLM passes through a 6-step governance pipeline before reaching the actual tool implementation. This ensures safety, auditability, and user control.

The 6-Step Pipeline

LLM returns tool_use block | [1] VALIDATE -----> Tool exists? Input valid? Mode allows it? | | | (reject if invalid) | [2] IGNORE CHECK --> IgnoreService: path blocked? | | | (deny if ignored/protected) | [3] APPROVAL ------> Write operation? Auto-approve config? | | | (show diff, wait for user) | [4] CHECKPOINT ----> GitCheckpointService: snapshot before write | [5] EXECUTE -------> BaseTool.execute(input, context) | [6] LOG ------------> OperationLogger: append to JSONL | v Return tool_result to LLM

BaseTool Interface

Every tool extends BaseTool:

abstract class BaseTool {
  abstract name: string;
  abstract isWriteOperation: boolean;
  abstract execute(
    input: Record<string, any>,
    context: ToolExecutionContext
  ): Promise<string>;

  formatSuccess(msg): string;  // wraps in <success> tags
  formatError(msg): string;    // wraps in <error> tags
  formatContent(msg): string;  // wraps in <content> tags
}

The isWriteOperation flag determines whether the tool needs approval and checkpoint creation.

Tool Repetition Detection

The pipeline maintains a 10-call sliding window that tracks each tool invocation as a composite key of toolName + JSON.stringify(input). After every tool execution, the window is checked for repeated patterns.

When the same composite key appears 3 or more times within the sliding window, the loop detection triggers:

  1. An error message is injected into the tool result: <error>Tool repetition loop detected. Aborting.</error>
  2. signalCompletion('aborted: tool repetition loop') is called, which terminates the current agent loop iteration.
  3. The user sees a notification that the task was aborted due to repetitive tool calls.

The sliding window is reset on mode switch -- when the agent switches from one mode to another (e.g., Ask to Agent), the repetition history is cleared. This prevents false positives when a new mode legitimately needs to call the same tools. The window is also reset at the start of each new conversation turn.

This mechanism prevents runaway loops where the LLM repeatedly calls the same tool with identical arguments (e.g., reading the same file in a loop, or retrying a failed search with the same query). It acts as a safety net complementing the consecutiveMistakeLimit on the AgentTask level.

ToolExecutionContext

Every tool's execute() method receives a ToolExecutionContext object as its second argument. This context provides the tool with access to agent-level capabilities without requiring direct coupling to the AgentTask or UI layer.

interface ToolExecutionContext {
  spawnSubtask: (description: string, mode: string) => Promise<string>;
  switchMode: (slug: string) => void;
  signalCompletion: (result: string) => void;
  askQuestion: (question: string, options?: string[]) => Promise<string>;
  updateTodos: (todos: TodoItem[]) => void;
  onApprovalRequired: (tool: string, input: Record<string, any>) => Promise<boolean>;
  onCheckpoint: (info: CheckpointInfo) => void;
  invalidateToolCache: () => void;
}

The context is constructed by the AgentTask and passed through the ToolExecutionPipeline. The same context object is shared across all tool calls within a single agent loop iteration, ensuring consistent state. The ContextExtensions pipeline mirrors all context fields so that tool wrappers and middleware can intercept calls.

Parallel Execution

When the LLM returns multiple tool_use blocks in a single response, the pipeline evaluates whether they can be executed in parallel or must run sequentially.

A tool is eligible for parallel execution only if it is in the PARALLEL_SAFE set -- a hardcoded allowlist of 13 read-only, side-effect-free tools:

const PARALLEL_SAFE = new Set([
  'read_file',
  'list_files',
  'search_files',
  'get_frontmatter',
  'get_linked_notes',
  'search_by_tag',
  'get_vault_stats',
  'get_daily_note',
  'web_fetch',
  'web_search',
  'semantic_search',
  'query_base',
  'open_note',
]);

The decision logic is straightforward:

  1. If the LLM returns only 1 tool call, it runs immediately (no parallelism needed).
  2. If the LLM returns 2+ tool calls and all of them are in the PARALLEL_SAFE set, they execute via Promise.all().
  3. If any tool in the batch is not in the PARALLEL_SAFE set, the entire batch runs sequentially in the order returned by the LLM.

Sequential fallback is deliberate: write operations may depend on each other's results (e.g., create a file then write to it), and the approval flow for write tools requires serial user interaction. Mixing parallel reads with serial writes in a single batch would create race conditions with the approval dialog.
