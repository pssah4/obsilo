UI Architecture

Obsilo's UI is built as an Obsidian ItemView registered in the right sidebar. The AgentSidebarView orchestrates message rendering, user input, autocomplete, and all interactive elements.

AgentSidebarView

Extends Obsidian's ItemView (view type: obsidian-agent-sidebar, display name: "Obsilo Agent").

Message Flow

User types message | v AutocompleteHandler checks for triggers (/, @) | v Message submitted | +-- Create AbortController +-- Render user message bubble +-- Instantiate AgentTask(api, tools, callbacks, mode, ...) +-- task.run(message, history) | | | +-- onText(chunk) --> Append to assistant bubble | +-- onThinking(text) --> Render thinking block | +-- onToolStart(name, in) --> Render tool card (pending) | +-- onToolResult(name, out)--> Update tool card (complete) | +-- onApprovalRequired() --> Show diff modal, await user | +-- onQuestion() --> Show question dialog | +-- onTodoUpdate() --> Update todo box | +-- onCheckpoint() --> Show undo bar | +-- onUsage() --> Update token display | +-- onComplete() --> Finalize, save to store | v Conversation persisted to ConversationStore

Rendering Components

Message Bubbles

User and assistant messages rendered as markdown (Obsidian's MarkdownRenderer.render()). Assistant bubbles append incrementally during streaming.

Tool Cards

Each tool call renders as a collapsible card showing tool name, badge (group color), input parameters, and result. Read-only tools with same group are merged into a single row.

Groupable tools: read_file, list_files, search_files, search_by_tag, get_vault_stats, get_linked_notes, open_note, get_daily_note, semantic_search.

Thinking Blocks

Extended thinking content rendered in a distinct collapsible block with muted styling.

Diff-Stats Badge

After write operations, a badge shows lines added/removed: +12 -3.

Token Usage

Displayed in the message footer: HH:MM · X,XXX in · Y,YYY out.

Chat Autocomplete

The AutocompleteHandler provides two trigger characters:

TriggerShowsResult
/Workflows + custom promptsInserts workflow slash command
@Vault files (VaultFilePicker)Attaches file reference

VaultFilePicker

Live search across all vault markdown files. Max 10 filtered results. Active note pinned at top. Supports keyboard navigation (arrow keys, enter, escape) and multi-select.

ToolPickerPopover

Session-level tool configuration. Groups: Built-In (with group checkboxes), MCP Servers (toggle), Skills (async-loaded), Workflows (radio-style). Changes persist for the session only.

HistoryPanel

Overlay panel showing past conversations. Rendered as a sliding overlay with a CSS transition (200 ms) from the left edge of the sidebar.

Date Grouping

Conversations are organized into four groups: Today, Yesterday, This Week, and Older. Within each group, entries are sorted in reverse-chronological order (newest first).

Interaction

  • Search filter — Substring match across conversation titles and content. Results update as the user types.
  • Restore & continue — Clicking an entry calls loadConversation(), which restores the full conversation history and resumes the session.
  • Delete — Individual entries can be removed from the store.

Persistence

All conversations are persisted by ConversationStore as JSON files on disk, one file per conversation.

Approval Flow

Two modal types for write operation approval:

  • DiffReviewModal — Shows full diff with syntax highlighting for file edits
  • ApproveEditModal — Simplified approve/reject for non-diff operations (create, delete, move)

Both return a Promise<boolean> that the pipeline awaits.

DiffReviewModal Detail

The DiffReviewModal operates in two distinct modes:

ModeContextActions
reviewPost-task review of changesPer-section: Keep / Undo / Edit. Global: Apply Selected, Keep All, Undo All
checkpointRead-only inspection of a past checkpointRestore button (reverts vault to that snapshot)

Semantic Section Grouping

Diffs are not shown as raw line-level patches. Instead, the modal groups changes into semantic sections based on Markdown structure: frontmatter, headings, code blocks, lists, callouts, tables, and paragraphs. Each section is independently reviewable.

Side-by-Side Diff with Context Collapse

Changes are rendered in a two-column layout (before / after). When more than 7 consecutive unchanged lines appear between changed regions, they are collapsed into a clickable expander to keep the diff focused on actual modifications.

Inline Edit Capability

In review mode, clicking the Edit action on a section opens an inline textarea pre-filled with the proposed content. The user can modify the text freely. Edited content is tracked separately and spliced into the correct position when the user applies the final selection.

Settings UI

The settings interface is built with Obsidian's PluginSettingTab and organized as 4 top-level tabs, each with its own sub-tabs:

Top-Level TabSub-Tabs
ProvidersModels, Embeddings, Web Search, MCP
Agent BehaviourModes, Auto-Approve, Loop, Memory, Rules, Workflows, Skills, Prompts
VaultSemantic Index, Ignored Paths
AdvancedInterface, Shell, Log, Debug, Backup

Router State

Navigation state is managed by three properties: activeTab, activeProvidersSubTab, activeAgentSubTab, and activeAdvancedSubTab. The method openAt(tab, subTab) supports deep-linking — other components can programmatically jump to a specific settings pane (for example, a "Configure provider" link in an error message opens the Providers → Models sub-tab directly).

Toolbar Elements

The chat toolbar sits between the message area and the input field. It contains 9 interactive elements:

  1. Mode button — Dropdown showing all available modes. Click to switch the active mode for the current conversation.
  2. Model button — Override the model on a per-conversation basis, independent of the global default.
  3. Tool picker (pocket-knife icon) — Opens the ToolPickerPopover to toggle individual tools, skills, workflows, and MCP servers.
  4. Web toggle (globe icon) — Enable or disable web tools (web_fetch, web_search) for the current session.
  5. Attach (paperclip icon) — Opens a native file picker. Selected files are processed through AttachmentHandler.
  6. @ trigger — Typing @ in the input opens the VaultFilePicker for vault file references.
  7. More options (ellipsis icon) — Context menu with additional actions (export, clear, settings shortcuts).
  8. Stop button — Visible only during task execution. Fires the AbortController signal to cancel the running agent loop.
  9. Send button — Triggers handleSendMessage(), which packages attachments, conversation history, and user text into the agent task.

AttachmentHandler

The AttachmentHandler manages file attachments in the chat input area.

Entry Points

MethodSourceBehaviour
processFile(File)Toolbar paperclip, drag-and-drop, clipboard pasteImages: base64-encoded. Text files: content wrapped as text blocks.
addVaultFile(TFile)@ mention pickerReads file content via vault.read() and attaches it as a vault reference.

Input Methods

  • Clipboard paste — Images pasted from clipboard are intercepted and processed automatically.
  • Drag-and-drop — Files dragged into the input area trigger a visual drop-zone indicator before processing.

Constraints and Rendering

Maximum file size is 10 MB. Attached files render as chips below the input field: images show a thumbnail preview, text files display a file-text icon with the filename.

Send Behaviour

When the user sends a message, the handler snapshots all pending attachments, clears the attachment area, and builds a ContentBlock[] array. Ordering: image blocks first, then text content blocks, then vault file blocks. This array is merged into the user message before being passed to the agent task.

For how to use the chat interface, see the Chat Interface user guide.
