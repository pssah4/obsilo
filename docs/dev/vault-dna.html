VaultDNA

VaultDNA is the plugin discovery system that automatically scans all installed Obsidian plugins, classifies their capabilities, and generates skill files that enable the agent to use any plugin's commands and APIs.

Plugin Discovery

Obsidian Plugin Start | v VaultDNA Scanner | +-- Enumerate all installed plugins (core + community) | +-- For each plugin: | +-- Read manifest (name, description, version) | +-- Discover commands (app.commands) | +-- Classify: FULL / PARTIAL / NONE | +-- API reflection (Tier 2, if available) | +-- Generate skill file | +-- Continuous sync (5s polling) | +-- Detect enable/disable/install/remove | +-- Regenerate affected skill files

Classification System

LevelCriteriaDescription
FULL3+ meaningful commandsPlugin fully usable by the agent
PARTIAL1-2 meaningful commandsLimited functionality available
NONE0 meaningful commandsPlugin not directly usable

UI-only commands are excluded from classification: toggle, show-, focus, settings, -panel, -sidebar, -pane.

API Discovery (Reflection)

For plugins that expose a public API via plugin.api, VaultDNA performs runtime reflection:

  • Enumerate all methods on the API object
  • Filter out blocked methods: constructor, execute, executeJs, render, register, unregister, onload, onunload, destroy, eval
  • Filter out private methods (starting with _)
  • Record method names and parameter signatures

Skill File Generation

Each plugin gets a Markdown skill file at .obsidian-agent/plugin-skills/{plugin-id}.skill.md:

---
name: Plugin Name
id: plugin-id
version: 1.2.3
classification: FULL
commands: 5
apiMethods: 3
---

# Commands
- command-id-1: "Command Name 1"
- command-id-2: "Command Name 2"
...

# API Methods
- methodA(param1, param2)
- methodB()

Settings Sanitization

When reading plugin settings for context, sensitive fields are automatically redacted:

  • Sensitive patterns: api-key, secret, password, token, credential, oauth
  • Internal state: lastSync, cache, version, __prefix
  • Size limits: strings truncated at 500 chars, arrays show 3-item preview, max depth 3
  • Total output cap: 8,000 characters

README Fetching

For community plugins, the scanner fetches README files from GitHub to enhance skill descriptions:

  • 7-day cache duration
  • Rate limited: 1 request/second
  • Max README size: 20,000 characters

Plugin Tools

ToolPurpose
execute_commandRun any Obsidian command by ID
enable_pluginEnable a disabled plugin
resolve_capability_gapFind plugins for a needed capability (active skills, then disabled plugins)
call_plugin_apiCall a method on a plugin's public API
execute_recipeRun a multi-step recipe (plugin + vault operations)

execute_command

Validates the given command_id against app.commands.commands before execution. If the command is not found, the tool suggests similar command IDs to help the agent self-correct. Execution is dispatched via executeCommandById(). On startup, the tool waits 500ms for new plugin command registrations before considering the command registry complete.

call_plugin_api

Uses a two-tier allowlist system to control which plugin methods the agent can invoke. Tier 1 is a built-in curated list of safe methods for well-known plugins (dataview.query, omnisearch.search, metaedit methods, etc.). Tier 2 uses dynamic discovery via runtime reflection on the plugin's API object, filtering out dangerous methods. Methods that are always blocked: execute, executeJs, render, and eval. All calls enforce a 10-second timeout and truncate the response to maxReturnSize.

execute_recipe

Runs multi-step recipes with 7 security layers: master toggle (global on/off), per-recipe toggle, parameter validation against declared schemas, no shell expansion via spawn() (not exec()), pipeline approval for write operations, process confinement, and audit trail logging. Built-in recipes include pandoc-pdf, pandoc-docx, and check-dependency. Binary paths are resolved at runtime using which/where to avoid hardcoded paths.

resolve_capability_gap

Performs a 3-stage search to find plugins matching a described capability. Stage 1: keyword-match against active skills. Stage 2: search disabled plugins. Stage 3: search archived/community plugins. Keyword extraction uses words of 3 or more characters, case-insensitive matching, with no stemming applied.

Built-in API Allowlist

The Tier 1 allowlist defines curated, pre-approved methods for well-known plugins. Each entry specifies the access level (read or write) and the maximum return size.

Continuous Sync

Polling every 5 seconds detects plugin enable/disable changes. On change, affected skill files are regenerated with a 3-second reclassification delay to allow Obsidian to fully initialize the plugin.

For how to use plugin skills, see Rules, Skills & Workflows.
