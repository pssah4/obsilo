System Prompt Architecture

The system prompt is assembled dynamically from 16 modular sections, each managed by a dedicated builder function in src/core/prompts/sections/. The orchestrator buildSystemPromptForMode() concatenates them in a fixed order.

Prompt Assembly

On every iteration, the system prompt is rebuilt with current context (time, active file, memory, tool definitions). This ensures the LLM always has up-to-date information.

buildSystemPromptForMode(mode, settings, vault, memory, tools, ...) | +-- [1] dateTime -- Current date and time +-- [2] vaultContext -- Vault name, active file, settings summary +-- [3] capabilities -- High-level agent capabilities +-- [4] memory -- User profile, projects, patterns, soul +-- [5] tools -- Mode-filtered tool definitions (JSON schema) +-- [6] pluginSkills -- VaultDNA compact skill list +-- [7] toolRules -- Usage rules for specific tools +-- [8] toolDecisionGuide -- Decision logic for tool selection +-- [9] objective -- 11-point task decomposition framework +-- [10] responseFormat -- Response structuring guidelines +-- [11] explicitInstr -- Best practices and anti-patterns +-- [12] securityBoundary -- "The vault is sacred" guardrails +-- [13] modeDefinition -- Active mode role definition +-- [14] customInstructions -- Global + mode-specific instructions +-- [15] skills -- Keyword-matched skill content +-- [16] rules -- Active vault/global rules | v Single concatenated system prompt string

The 16 Sections

1. dateTime

Injects the current date, time, and timezone so the LLM can make time-aware decisions (e.g., daily notes, deadlines).

2. vaultContext

Provides vault name, currently active file path, a summary of relevant settings, and workspace context.

3. capabilities

High-level overview of what the agent can do: read/write files, search, web access, memory, MCP tools, etc. Gives the LLM a mental model of its own abilities.

4. memory

Injects up to 4000 characters from 5 memory files (800 chars each max): soul.md, user-profile.md, projects.md, patterns.md, learnings.md. Each wrapped in XML tags like <agent_identity>, <user_profile>.

Each file is read from .obsidian-agent/memory/ and truncated to a hard limit of 800 characters. If a file exceeds this limit, it is cut at the nearest word boundary and an ellipsis is appended. The total budget for the memory section is 5 x 800 = 4000 characters.

The XML wrapping strategy uses semantic tag names per file:

FileXML TagPurpose
soul.md<agent_identity>Agent persona and personality traits
user-profile.md<user_profile>User preferences, language, workflows
projects.md<projects>Active projects and their context
patterns.md<patterns>Architecture patterns and conventions
learnings.md<learnings>Accumulated insights from past sessions

Note: knowledge.md is not included in the system prompt. Knowledge files are intended for the semantic search index and are retrieved on-demand via semantic_search, not injected into every prompt. This is a deliberate design choice to keep the memory section within its token budget. Files that do not exist are silently skipped -- no placeholder or empty tag is emitted.

5. tools

The complete JSON schema definitions for all tools accessible in the current mode. Tools are filtered by mode's tool group configuration. This is typically the largest single section in the prompt.

Each tool definition is emitted as a JSON schema object containing the tool's name, description, and input_schema (with type, properties, and required fields). This follows the standard Anthropic/OpenAI function-calling format so the LLM can generate valid tool_use blocks.

The set of tools included is determined at prompt assembly time by ModeService.getEffectiveToolNames(). This method resolves the active mode's toolGroups array (e.g., ["edit", "web", "agent"]) into a flat list of individual tool names using expandToolGroups(), then filters the ToolRegistry to only include matching tools. Custom modes can also specify customTools to add or remove individual tools beyond their group defaults.

With all 42 tools enabled (as in Agent mode), this section alone can consume 8,000-12,000 tokens. Read-only modes like Ask mode include only the read and agent groups, reducing this significantly.

6. pluginSkills

Compact listing of VaultDNA-discovered plugin skills. One line per plugin with available commands and API methods.

7. toolRules

Specific usage rules for tools that constrain and guide the LLM's tool selection behavior. These are hard rules the agent must follow, not suggestions. Examples include:

  • "Use search_files before read_file for unknown paths" -- When the user asks about a file or concept but the exact path is unknown, the agent must first search to discover the path rather than guessing and risking a hallucinated path.
  • "Prefer semantic_search for conceptual queries" -- When the user asks a meaning-based question (e.g., "Where did I write about project goals?"), the agent should use the vector-based semantic search rather than keyword-based search_files.
  • "Always show diffs before write operations" -- Before using write_to_file or apply_diff, the agent should explain what changes will be made so the user can review them in the approval dialog.
  • "Use list_files to orient before deep-reading" -- For broad exploration tasks, list directory contents first to understand the structure before reading individual files.

Tool rules are static strings defined in the prompt builder. They are always included regardless of mode, ensuring consistent behavior across Ask, Agent, and custom modes.

8. toolDecisionGuidelines

A structured decision tree that helps the LLM choose between tools with overlapping capabilities. While section [7] provides hard rules, section [8] provides nuanced guidance for ambiguous situations.

The key decision paths include:

ScenarioRecommended ToolRationale
Know the exact file pathread_fileDirect read is fastest, no search overhead
Know a keyword, need file pathssearch_filesRegex-based content search across vault
Conceptual/meaning-based querysemantic_searchVector similarity finds related content even without keyword match
Need structural context (links, tags)get_linked_notes / search_by_tagGraph-aware queries that follow Obsidian's link structure
Explore an unfamiliar directorylist_filesDirectory listing before targeted reads
Tabular data operationsquery_baseSQL-like queries on Obsidian Bases

This section also instructs the LLM to prefer read-only tools when the task only requires information retrieval, and to escalate to write tools only when modification is explicitly requested or clearly implied by the user's intent.

9. objective

An 11-point task decomposition framework that structures how the agent approaches complex tasks. The full framework instructs the agent to follow these steps:

  1. Understand the request -- Parse the user's intent; identify explicit and implicit requirements.
  2. Assess current state -- Use read tools to understand what already exists before making changes.
  3. Plan the approach -- Break the task into discrete, ordered sub-steps before executing anything.
  4. Validate assumptions -- Confirm file paths, structures, and data formats before acting on them.
  5. Execute incrementally -- Make one logical change at a time; avoid large batch operations.
  6. Verify each step -- After each write operation, confirm the result matches expectations.
  7. Handle errors gracefully -- If a tool call fails, diagnose the cause before retrying with corrections.
  8. Maintain context -- Reference previous findings and decisions; do not re-read files unnecessarily.
  9. Communicate progress -- Keep the user informed of what was done, what was found, and what comes next.
  10. Know when to ask -- If the task is ambiguous or risky, use ask_followup_question rather than guessing.
  11. Signal completion -- Use attempt_completion with a clear summary when the task is fully done.

This framework is the core behavioral contract between the system prompt and the LLM. It ensures the agent acts methodically rather than impulsively, reducing hallucinated paths and unnecessary write operations.

10. responseFormat

Guidelines for how the agent should structure its responses: use markdown, be concise, cite sources, format code blocks.

11. explicitInstructions

Best practices and anti-patterns: don't apologize, don't hallucinate file paths, don't repeat failed operations, acknowledge limitations.

12. securityBoundary

The "vault is sacred" guardrail: never delete files without explicit permission, always show diffs, respect .obsidian-agentignore, no destructive operations on system files.

13. modeDefinition

The role definition for the currently active mode. For Ask mode: "You are a read-only assistant..." For Agent mode: "You are a full-featured agent with write access..." Custom modes inject their own role text.

14. customInstructions

User-defined instructions from two sources: global custom instructions (apply to all modes) and per-mode custom instructions. Both are plain text injected verbatim.

15. skills

Content from matched skill files. Skills are activated by keyword matching against the user's message. When a skill matches, its full content is injected into this section.

The matching algorithm uses a bag-of-words approach:

  1. The user's message is tokenized into lowercase words.
  2. Tokens shorter than 3 characters are discarded (prepositions, articles, etc.).
  3. Each skill file defines a keywords array in its YAML frontmatter.
  4. A skill matches if any of its keywords appears as a token in the user's message.
  5. No stemming is applied -- keywords must match exactly (e.g., "canvas" matches but "canvases" does not).

When a skill matches, its content (everything below the YAML frontmatter) is injected into the prompt, capped at 4000 characters per skill. If multiple skills match, all are included, each wrapped in <skill name="..."> XML tags. Skill files are stored in .obsidian-agent/skills/ as Markdown files with YAML frontmatter containing the keywords and enabled fields.

16. rules

Active rules from .obsidian-agent/rules/ and global rules. Each enabled rule file's content is injected wrapped in <rule> tags.

Rule files are Markdown files stored in the vault-local directory .obsidian-agent/rules/. Each rule file has a YAML frontmatter with an enabled boolean toggle. Only rules with enabled: true are included in the prompt. Global rules (from the plugin's global settings directory) follow the same format and are merged with vault-local rules.

Each individual rule file is capped at 50,000 characters. Content beyond this limit is truncated. The rule content is wrapped in <rule name="filename">...</rule> XML tags, similar to the skill injection pattern. Rules are always injected regardless of the user's message content (unlike skills, which require keyword matching).

Typical use cases for rules include: coding conventions, naming standards, vault organization policies, and project-specific instructions that should apply to every conversation in the vault.

Token Budget

The assembled system prompt must fit within the model's context window alongside the full conversation history (user messages, assistant messages, and tool results). Since the context window is finite (typically 128k or 200k tokens depending on the provider), the system prompt competes directly with conversation length.

The largest sections by token count are typically:

SectionTypical SizeNotes
[5] tools8,000-12,000 tokensScales with number of enabled tool groups
[4] memory1,000-2,000 tokensHard-capped at 4000 chars (5 files x 800)
[15] skills0-3,000 tokensOnly present when keywords match; 4000 chars/skill
[16] rules0-5,000+ tokensDepends on number and size of active rule files
[9] objective~500 tokensFixed static content
All others~200-400 tokens eachRelatively small static or semi-static sections

A fully loaded system prompt (Agent mode, all tools, memory populated, rules and skills active) can reach 15,000-25,000 tokens. This is why read-only modes and subtask prompts deliberately exclude sections -- to preserve context window budget for the actual conversation.

Subtask Mode

When a child task is spawned via new_task, it receives a deliberately leaner prompt to maximize context window budget for the subtask's focused work. The subtask prompt includes only the sections essential for tool execution and safety.

Included sections:

SectionReason for Inclusion
[1] dateTimeTime-awareness is always needed
[2] vaultContextSubtask needs to know vault name and active file
[3] capabilitiesAgent must know what it can do
[4] memoryUser preferences and project context apply to subtasks
[5] toolsTool definitions are required for tool use
[7] toolRulesSafety rules must always apply
[12] securityBoundaryGuardrails are non-negotiable in any context

Excluded sections and rationale:

SectionReason for Exclusion
[6] pluginSkillsSubtasks do not discover plugins independently
[8] toolDecisionGuidelinesSubtasks have a clear objective; no ambiguity in tool choice
[9] objectiveReplaced by the parent's task description
[10] responseFormatSubtask output goes to parent, not to the user
[11] explicitInstructionsReduces prompt size; parent handles user-facing concerns
[13] modeDefinitionSubtask mode is defined by the parent's new_task call
[14] customInstructionsNot inherited to keep subtask focused on its specific task
[15] skillsNot inherited; subtask operates on explicit instructions only
[16] rulesNot inherited to reduce prompt size and avoid rule conflicts

The parent's task description is appended as a dedicated <task> section, replacing the objective framework. This keeps subtask prompts at roughly 40-60% of the size of a full prompt.

For mode definitions and custom instructions, see the Modes user guide.
