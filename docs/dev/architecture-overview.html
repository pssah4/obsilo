Architecture Overview

Obsilo is an AI agent built as an Obsidian plugin. It runs entirely in the Electron renderer process, with no backend server, combining an agentic loop with a governed tool execution pipeline and a modular prompt system.

System Context

Obsilo connects three external boundaries: the Obsidian Vault (local filesystem), LLM API providers (Anthropic, OpenAI, Google Gemini, Ollama, Azure, OpenRouter, LM Studio, or custom), and optionally MCP servers for extended tool access.

Key Architectural Decisions

  1. Central tool execution pipeline — Every tool call passes through a 6-step governance flow: validate → ignore check → approval → checkpoint → execute → log. Why: Ensures no tool can bypass safety checks.
  2. Fail-closed approval — If no approval callback is registered, write operations are denied by default. Why: Better to deny a safe operation than allow an unsafe one.
  3. Shadow git repository — isomorphic-git maintains a hidden repo for checkpoint/restore without requiring system git. Why: No external git dependency, works on any OS.
  4. Mode-based tool filtering — Each mode defines which tool groups are accessible, preventing unintended operations. Why: Reduces attack surface by limiting tool access per context.
  5. Semantic index — Vectra HNSW vector store with configurable embedding model (API or local via Ollama). Why: Semantic search across the entire vault.
  6. Sliding-window repetition detection — Prevents infinite tool loops by tracking recent calls in a 10-call window. Why: Bounded memory prevents infinite loops without false positives.
  7. Multi-provider API adapter — Unified ApiHandler interface normalizes Anthropic, OpenAI, Ollama, Azure, and custom providers. Why: Users choose their preferred LLM without code changes.
  8. 3-tier memory architecture — Chat history → session extraction → long-term memory promotion via LLM pipeline. Why: Balances detail (chat) with efficiency (promoted facts) across time.
  9. VaultDNA plugin discovery — Runtime reflection scans installed plugins, generates skill files automatically. Why: Plugin ecosystem is Obsidian's strength; VaultDNA bridges it to the agent.
  10. Hybrid search with RRF — Semantic + BM25/TF-IDF combined via Reciprocal Rank Fusion (k=60). Why: Semantic handles conceptual queries; BM25 handles exact matches; RRF combines both.
  11. Agent Skill Mastery (3 layers) — Rich tool descriptions with examples, procedural recipes with keyword-first matching, and episodic task memory with auto-promotion. Why: The agent improves over time by learning from successful task executions.

Document Parser Pipeline

Task Extraction Pipeline

A deterministic, regex-based post-processing pipeline that scans agent responses for unchecked markdown checkboxes (- [ ]) and converts them into structured task notes with frontmatter. ADR-026/027.

Agent Response Text | TaskExtractor.scan() [src/core/tasks/TaskExtractor.ts] | regex: /^[\t ]*[-*]\s\[ \]\s+(.+)$/gm | extracts: text, @assignee, due: YYYY-MM-DD v TaskItem[] (parsed tasks) | TaskSelectionModal [src/ui/TaskSelectionModal.ts] | user selects which tasks to create v TaskNoteCreator.createNotes() [src/core/tasks/TaskNoteCreator.ts] | smart title generation (natural break points) | 10-property YAML frontmatter | unique path generation (slug + dedup) v Task notes in configurable folder

Chat-Linking

Bidirectional linking between vault notes and agent conversations. Two mechanisms:

  1. Manual link — User clicks the link button in chat history to stamp the active note's frontmatter with a Chats array entry containing an obsidian://obsilo-chat?id=... deep link.
  2. Auto-link on finalize — When a conversation ends, finalizeConversation() generates a semantic title (via configurable LLM model) and stamps any notes that were actively edited during the session.

Frontmatter format:

Chats:
  - "[Conversation Title](obsidian://obsilo-chat?id=abc123)"

Duplicate detection prevents the same conversation from being linked twice. Deep links open directly in the Obsilo sidebar.
