Agent Loop

The AgentTask class is the central conversation loop. It manages the iterative cycle of sending messages to the LLM, processing tool calls, and accumulating conversation history until a termination condition is reached.

Iteration Flow

Each iteration follows 13 sequential steps:

  1. Abort check — If abortSignal is set, exit immediately.
  2. Mode switch — Apply any pending mode switch from the previous iteration.
  3. Rate limiting — Delay by rateLimitMs if configured.
  4. Power steering — Every N iterations, inject a system reminder with the current task context.
  5. Rebuild prompt cache — Regenerate the system prompt with updated vault context, memory, and tool definitions.
  6. Stream LLM response — Send conversation history to the API and process the response stream.
  7. Accumulate tool calls — Collect tool_use blocks from the stream into complete objects.
  8. Execute tools — Process all tool calls through the ToolExecutionPipeline. Tools marked PARALLEL_SAFE run concurrently.
  9. Append results — Add tool results to conversation history as tool_result content blocks.
  10. Context condensing — If token estimate exceeds threshold, summarize older messages (keep first + last 4).
  11. Completion check — If attempt_completion was called or no tools were invoked, end the loop.
  12. Repetition detection — Check the 10-call sliding window for repeated tool patterns.
  13. Iteration counter — Increment and check against maxIterations.

Parallel Tool Execution

When the LLM returns multiple tool calls in a single response, tools in the PARALLEL_SAFE set run concurrently via Promise.all(). Write operations and tools with side effects execute sequentially.

The complete PARALLEL_SAFE set contains 13 read-only tools:

  1. read_file
  2. list_files
  3. search_files
  4. get_frontmatter
  5. get_linked_notes
  6. search_by_tag
  7. get_vault_stats
  8. get_daily_note
  9. web_fetch
  10. web_search
  11. semantic_search
  12. query_base
  13. open_note

Parallel execution triggers only when all tool calls in the batch belong to this set and the batch contains more than one tool call. If any single tool in the batch is not parallel-safe, the entire batch executes sequentially.

Multi-Agent Orchestration

The new_task tool allows an agent to spawn a child AgentTask that runs as a subtask within the parent loop.

Nesting Depth

Maximum nesting depth is configurable via maxSubtaskDepth (default: 2). Each child task is created with depth = parent.depth + 1. If a child attempts to spawn beyond the limit, the tool returns an error instead of creating a new task.

Result Propagation

When a child task completes (via attempt_completion or natural end), its result text is returned to the parent as a tool_result content block. The parent continues its own loop with the child's output as context.

Context Condensing

As a conversation grows, the token count approaches the model's context window limit. Context condensing automatically compresses older messages to prevent truncation or API errors.

Trigger

Condensing triggers when the estimated token count exceeds condensingThreshold (default: 80%) of the active model's maximum context window size. The estimate is calculated before each LLM call.

Strategy

The condensing algorithm preserves the first message (original user request) and the last 4 messages (recent context). Everything in between is summarized by sending the intermediate messages to the LLM with a summarization prompt. The resulting summary is inserted as a system message, replacing the original intermediate messages.

UI Notification

After condensing completes, the onContextCondensed() callback notifies the UI, which displays a badge in the message area indicating that context was compressed.

Subtask Limitation

Context condensing is disabled for subtasks. Child tasks spawned via new_task start with a fresh, empty conversation history and are expected to complete within a single context window.
