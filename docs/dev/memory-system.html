Memory Architecture

Obsilo implements a 3-tier memory system that enables the agent to learn from conversations: chat history for short-term, session summaries for medium-term, and promoted facts for long-term persistence.

3-Tier Architecture

Tier 1: Chat History (ConversationStore) Full conversations as JSON, per session | (after conversation ends) v Tier 2: Session Summaries (SessionExtractor) LLM-generated summary, one per conversation Semantically searchable via MemoryRetriever | (async promotion via ExtractionQueue) v Tier 3: Long-Term Memory (LongTermExtractor) Facts promoted into persistent Markdown files user-profile.md, projects.md, patterns.md, soul.md, learnings.md

Memory Files

Storage: .obsidian/plugins/obsidian-agent/memory/

FileXML TagPurpose
soul.md<agent_identity>Agent personality, language, values, anti-patterns
user-profile.md<user_profile>User preferences, communication style
projects.md<active_projects>Current projects and contexts
patterns.md<behavioral_patterns>Learned interaction patterns
learnings.md<task_learnings>Task execution insights
knowledge.md(not in prompt)On-demand via semantic search only

Limits: 800 characters per file, 4000 total in system prompt.

ExtractionQueue

Persistent FIFO queue surviving Obsidian restarts. Queue file: .obsidian/plugins/obsidian-agent/pending-extractions.json.

Queue items: conversationId, transcript, title, queuedAt, type ('session' | 'long-term').

Two async LLM calls per conversation: session extraction + long-term promotion.

Memory Context Injection

Each memory file is injected into the system prompt wrapped in its XML tag. The buildMemoryContext() method reads all files, truncates to limits, and concatenates.

OnboardingService

5-step setup flow for new users. Creates default soul.md with name "Obsilo", language "Deutsch", and anti-patterns.

For how memory works from a user perspective, see Memory & Chat History.
