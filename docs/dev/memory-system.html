Memory Architecture

Obsilo implements a 3-tier memory system that enables the agent to learn from conversations: chat history for short-term, session summaries for medium-term, and promoted facts for long-term persistence.

3-Tier Architecture

Tier 1: Chat History (ConversationStore) Full conversations as JSON, per session | (after conversation ends) v Tier 2: Session Summaries (SessionExtractor) LLM-generated summary, one per conversation Semantically searchable via MemoryRetriever | (async promotion via ExtractionQueue) v Tier 3: Long-Term Memory (LongTermExtractor) Facts promoted into persistent Markdown files user-profile.md, projects.md, patterns.md, soul.md, learnings.md

Memory Files

Storage: .obsidian/plugins/obsilo-agent/memory/

FileXML TagPurpose
soul.md<agent_identity>Agent personality, language, values, anti-patterns
user-profile.md<user_profile>User preferences, communication style
projects.md<active_projects>Current projects and contexts
patterns.md<behavioral_patterns>Learned interaction patterns
learnings.md<task_learnings>Task execution insights
knowledge.md(not in prompt)On-demand via semantic search only

Limits: 800 characters per file, 4000 total in system prompt.

ExtractionQueue

Persistent FIFO queue surviving Obsidian restarts. Queue file: .obsidian/plugins/obsilo-agent/pending-extractions.json.

Queue items: conversationId, transcript, title, queuedAt, type ('session' | 'long-term').

Two async LLM calls per conversation: session extraction + long-term promotion.

Full Extraction Pipeline

  1. The conversation transcript is packaged as a PendingExtraction with type='session'.
  2. ExtractionQueue.enqueue() serializes the item and writes it to pending-extractions.json.
  3. processQueue() picks the next item from the front of the queue.
  4. SessionExtractor generates a 7-section summary via LLM call:
    • Summary -- what the conversation accomplished
    • Decisions -- choices made during the session
    • User Preferences Observed -- behavioral signals from the user
    • Task Outcome -- success/failure/partial result
    • Tool Effectiveness -- which tools worked well or poorly
    • Learnings -- insights for future interactions
    • Open Questions -- unresolved items
  5. Output format: YAML frontmatter + Markdown body, hard-capped at <400 words.
  6. The summary is saved to sessions/{conversationId}.md and indexed in the semantic index with source='session'.
  7. If autoUpdateLongTerm is enabled in settings: the queue automatically enqueues a follow-up item with type='long-term'.
  8. LongTermExtractor reads the current memory files and identifies facts worth promoting from the session summary.
  9. Output: JSON with { file, action, section, content } -- where action is either 'append' (add new fact) or 'replace' (update existing section).

Processing behavior: The queue is a sequential processor with a 2-second delay between items to avoid rate-limit pressure. Crash recovery is built in: an item is only dequeued from pending-extractions.json after successful processing, so if Obsidian crashes mid-extraction the item will be retried on next startup.

Memory Context Injection

Each memory file is injected into the system prompt wrapped in its XML tag. The buildMemoryContext() method reads all files, truncates to limits, and concatenates.

buildMemoryContext() Details

  • Reads all 5 memory files: soul.md, user-profile.md, projects.md, patterns.md, learnings.md. Note that knowledge.md is not included -- it is only accessible on-demand via semantic search.
  • Each file is truncated to a maximum of 800 characters.
  • Total budget for all memory context: 4000 characters.
  • Each file's content is wrapped in a descriptive XML tag:
    • <agent_identity> -- soul.md
    • <user_profile> -- user-profile.md
    • <active_projects> -- projects.md
    • <behavioral_patterns> -- patterns.md
    • <task_learnings> -- learnings.md
  • The concatenated XML block is injected as system prompt section [4], after mode instructions and before tool definitions.

MemoryRetriever

The MemoryRetriever provides on-demand access to past session summaries for agent tools that need to recall previous conversations.

  • Primary strategy -- semantic search: Calls semanticIndex.searchSessions(query, topK) with a source='session' filter. This finds the most relevant past sessions based on vector similarity to the current query.
  • Fallback strategy -- recency-based: If the semantic index is unavailable or returns no results, falls back to loading the most recent N session files sorted by modification date.
  • Output format: Returns an XML block with a <relevant_sessions> wrapper containing one entry per matched session. Each session summary is truncated to 600 characters to fit within context budgets.
  • Usage: Called by agent tools (e.g., recall_memory) when the LLM needs to reference information from past interactions that is not present in the current conversation history or long-term memory files.

OnboardingService

Interactive 8-step setup flow for new users. Initializes the agent's identity and the user's preferences on first launch.

8-Step Onboarding Flow

  1. Greeting -- Welcome message introducing the agent and the onboarding process.
  2. Naming -- The user chooses a name for the agent (default: "Obsilo").
  3. Backup import -- Optional import of a previous memory backup to restore agent state.
  4. Language selection -- The user selects their preferred language for all agent interactions.
  5. Vault usage description -- The user describes how they use their Obsidian vault (personal notes, research, work, etc.). Stored in projects.md.
  6. Tone preferences -- The user indicates their preferred communication style (formal, casual, concise, etc.). Stored in user-profile.md.
  7. Permission preset selection -- The user picks one of three governance presets:
    • "Freie Hand" (permissive) -- the agent can act freely with minimal confirmations
    • "Ausgewogen" (balanced) -- standard confirmation prompts for destructive actions
    • "Vorsichtig" (restrictive) -- the agent asks for confirmation before most tool executions
  8. Completion -- Settings are applied and soul.md is written with the configured identity.

Interaction rules: The onboarding always uses text responses before tool calls and calls ask_followup_question after each step (except the final completion step) to advance to the next phase.

Soul Identity

The agent's core identity is defined in soul.md, the first memory file loaded into every system prompt.

  • File: .obsidian/plugins/obsilo-agent/memory/soul.md
  • Default language: Determined during the onboarding flow (step 4). All subsequent agent responses use this language unless overridden.
  • Contains: Personality description, core values, and explicit anti-patterns.
  • Values (priority order):
    • Usefulness over politeness -- the agent prioritizes giving genuinely helpful answers over being agreeable
    • Honesty -- the agent admits uncertainty rather than guessing
    • Respect -- the agent treats the user as a competent peer
    • Learn from mistakes -- the agent acknowledges and corrects its errors
  • Anti-patterns (explicitly forbidden):
    • No empty phrases or filler ("Sure!", "Great question!")
    • No unnecessary apologies
    • No emojis in any output
For how memory works from a user perspective, see Memory & Chat History.
