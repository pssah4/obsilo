Mode System

Modes define the agent's role, accessible tools, and behavioral constraints. The ModeService resolves modes from a three-tier hierarchy and filters the tool registry accordingly.

Mode Resolution

Three-tier priority: built-in modes → global modes (~/.obsidian-agent/modes.json) → vault modes (per-vault settings). Vault modes with the same slug override built-in modes, allowing full customization per vault.

The resolution chain is: built-in → global → vault. getEffectiveToolNames() applies user overrides using intersection only — users can restrict tools but never escalate beyond what the mode defines. switchMode(slug) persists the selection to settings.currentMode. If the requested slug is invalid or not found, the system falls back to the ask mode.

Built-in Modes

ModeSlugTool GroupsRole
Askaskread, vault, agentRead-only Q&A assistant
Agentagentread, vault, edit, web, agent, mcp, skillFull-access agent with write capabilities

Custom Modes

Custom modes are defined using the ModeConfig interface. Each mode specifies a unique slug, the tools it can access, and the behavioral role definition injected into the system prompt.

interface ModeConfig {
  slug: string;
  name: string;
  icon?: string;
  description?: string;
  roleDefinition: string;
  toolGroups: string[];     // e.g. ['read', 'vault', 'web', 'agent']
  customInstructions?: string;
  modelOverride?: string;   // per-mode model selection
  mcpWhitelist?: string[];  // restrict MCP server access
  source: 'builtin' | 'global' | 'vault';
}

Users can create custom modes with specific tool access, custom role definitions, and per-mode model overrides. The source field indicates where the mode was defined: built-in modes ship with the plugin, global modes are stored in the user home directory, and vault modes live in per-vault settings.

Tool Group Mapping

expandToolGroups() resolves group names to individual tool names. Groups are hierarchical: read is a subset of vault, which is a subset of edit.

Mode Switching

The switch_mode tool allows the LLM to request a mode change during a conversation. The tool only registers the request — the actual switch is applied at the start of the next agent loop iteration, ensuring the current iteration completes with its original tool set.

The tool repetition detector resets its counters on mode switch, preventing false positives when the new mode legitimately re-uses similar tool patterns.

Rules System

Rules stored at .obsidian-agent/rules/. Each .md file is a rule with on/off toggle. Max file size: 50,000 chars.

Active rules injected into system prompt section 16 wrapped in <rule> tags.

Workflows

Stored at .obsidian-agent/workflows/. Slug derived from filename. Activated via /slug slash commands.

Workflow content wrapped in <explicit_instructions type="{slug}"> tags.

Skills

Keyword-matched content injected into system prompt section 15. Skills auto-activate when user message contains matching keywords.

The SkillsManager reads skill files from the directory .obsidian-agent/skills/{name}/SKILL.md. Each SKILL.md file has YAML frontmatter with name and description fields, followed by the instructions body.

Keyword matching uses a bag-of-words approach: tokens of 3 or more characters are extracted from the user message, with no stemming applied. If any keyword matches a skill's registered keywords, that skill's content is included. Content is capped at 4,000 characters per skill and wrapped in <available_skills> XML tags in the system prompt.

Power Steering

Power Steering injects a mode reminder into the conversation every N agent iterations, reinforcing the active mode's role definition. The frequency is configurable (setting: powerSteeringFrequency); a value of 0 disables the feature entirely.

The reminder is skipped on iteration 0 (the initial prompt) and injected at the start of every N-th iteration thereafter. The injected message follows this format:

[Power Steering Reminder]

You are operating in **{mode}** mode.

{roleDefinition}

Continue the task.

This mechanism prevents the LLM from drifting away from its assigned role during long multi-step tasks.

Support Prompts

Support prompts are quick-action templates that prepend structured instructions to the user's input. Four built-in templates are provided:

TemplatePurpose
ENHANCEImprove or refine the selected text
SUMMARIZEGenerate a concise summary
EXPLAINExplain the content in detail
FIXIdentify and fix issues

Each template supports two variables: ${userInput} (the user's message or selection) and ${activeFileHint} (the currently active file path, if available). Users can override the built-in templates or add custom support prompts through settings.

For how to use and create modes, see Modes user guide.
