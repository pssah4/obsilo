Modes

Switch between modes to get the right kind of help for each task — from a read-only question-answering assistant to a fully autonomous agent that can create, edit, and organize your notes. You can also build your own custom modes with their own persona, tool access, and instructions.

Built-in Modes

Ask Mode

Ask is a read-only knowledge assistant. Use it to explore your vault, get synthesized answers from your notes, and find connections — without any risk of changing your files.

Try: "What do I know about machine learning?" / "Find notes I haven't updated in 30 days" / "What links to my Project X note?"

Access: read, vault intelligence, and semantic search tools. No write access.

Agent Mode

Agent is a fully autonomous agent with access to all tools. It reads, writes, edits, browses the web, and can spawn sub-agents to parallelize complex tasks.

Try: "Create a weekly review template" / "Research X on the web and summarize it to a new note" / "Add a 'status' frontmatter field to all notes in my Projects folder"

Access: All tool groups — read, vault, edit, web, skill, MCP, and agent control.

Which mode should I use? Start with Ask for exploration and questions. Switch to Agent when you want the AI to take action — create files, edit content, or run complex workflows.

Custom Modes

Create your own modes in Settings → Agent Behaviour → Modes. Custom modes can be:

  • Vault-scoped — stored in this vault's plugin settings, only available in this vault
  • Global — stored at ~/.obsidian-agent/modes.json, available in all vaults

Creating a Custom Mode

Click New Mode in the Modes settings tab. Configure:

FieldDescription
SlugURL-safe identifier, e.g. researcher or daily-writer
NameDisplay name shown in the mode selector, e.g. "Researcher"
IconLucide icon name, e.g. search, pen, book
DescriptionShort description shown in the mode selector tooltip
Role DefinitionCore system prompt for this mode — defines the agent's persona and behavior
Custom InstructionsUser-editable additional instructions appended to the role definition
Tool GroupsWhich tool categories are available in this mode
ScopeVault (this vault only) or Global (all vaults)

Example Custom Modes

Researcher

Slug: researcher
Tool Groups: read, vault, web, agent
Role Definition:
You are a research assistant focused on gathering and synthesizing
information. Use semantic_search to find existing knowledge first,
then use web_search and web_fetch to fill gaps with external information.
Always cite your sources and cross-reference with existing vault notes.

Daily Writer

Slug: daily-writer
Tool Groups: read, vault, edit, agent
Role Definition:
You are a writing assistant for daily journaling and reflection.
Focus on: reading today's daily note, summarizing recent notes,
drafting new entries. Write in a personal, reflective tone.
Use get_daily_note first to understand the current context.

Per-Mode Overrides

Model Override

Each mode can use a different AI model. Set a model override in Settings → Modes → (edit mode) → Model. When set, this model is used whenever the mode is active, overriding the global default.

Use case: Use a fast, cheap model (e.g., Claude Haiku or GPT-4o mini) for Ask mode, and a more capable model (e.g., Claude Sonnet) for Agent mode.

Tool Overrides

Restrict a mode to a specific subset of tools within its tool groups. For example, give Agent mode all edit tools except delete_file. Configured in Settings → Modes → (edit mode) → Tool Overrides.

MCP Server Whitelist

Limit which MCP servers a mode can access. Useful for restricting a sensitive mode from accessing powerful external tools. Set in Settings → Modes → (edit mode) → Allowed MCP Servers.

Forced Skills

Specify skills that are always injected into the system prompt for this mode, regardless of keyword matching. Useful for skills that should always be active in a mode (e.g., always inject a "meeting-notes" skill in a "meeting" mode).

Forced Workflow

Attach a workflow that runs automatically for every message in this mode (unless the message starts with /). The workflow content is prepended as explicit instructions to each message.

Multi-Agent Workflows

In Agent mode, the new_task tool lets the agent spawn sub-agents. Sub-agents run with a fresh conversation context in the specified mode:

  • new_task('ask', 'message') — spawn a read-only sub-agent
  • new_task('agent', 'message') — spawn a full-capability sub-agent

Sub-agents cannot see the parent conversation — pass all necessary context in the task message.

PatternWhen to use
Orchestrator-WorkerDecompose a large task into independent subtasks, each handled by a worker sub-agent
Prompt ChainingSequential steps where each agent's output feeds the next (research → draft → review)
Evaluator-OptimizerGenerate content, evaluate it, then refine until it meets quality criteria
RoutingRoute different subtasks to the appropriate mode (reads to Ask, writes to Agent)
